## 2. Workflow Rules (LOCKED IN — do not deviate)

These are rules Eric and Claude have settled over many sessions. Breaking them creates real problems.

**Operator-driven execution.** Eric runs every command locally on his Mac. Claude has no filesystem or shell access to Eric's machine. Claude proposes commands; Eric pastes them and pastes results back.

**Git discipline.**
- Never `git commit -a`. Never `git add .`. Always explicit file paths in `git add`.
- Single-line commit messages. No multi-line bodies.
- Push to `main` directly after each commit.

**Shell safety.**
- Never put `#` comments inside command blocks — zsh history mechanics can trip.
- Never write `!` in shebangs or strings inside pasted scripts — zsh history expansion fires.
- For Python scripts: always use heredoc with **quoted** `'EOF'` to prevent shell expansion of `$`, backticks, etc.
- For one-off Python: `python3 << 'EOF' ... EOF`. For scripts saved to disk: `cat > scriptname.py << 'EOF' ... EOF`.

**Smoke testing.**
- Cold restart for smoke tests (kill and re-run, not hot reload).
- Eric is pragmatic about purely mechanical changes (e.g., sed sweeps, field collapses) — he commits without explicit smoke for those, trusting the diff. For UX-affecting changes, smoke first.

**Production safety.**
- Never delete the Firestore production index `transactions: createdAt DESC + type ASC`. It's essential.
- Never push migration scripts (PHASE_*.py at repo root, patch_*.py) into git — they're operational, not source.

**Investigation principle.**
- Investigate before guessing. Don't assume file contents or symbol availability.
- Surface decisions to Eric, don't pre-pick. Eric chooses; Claude executes.
- If asked for plain English: no jargon.
- If a check fails or a search returns unexpected results, stop and investigate before forcing through.

**Tone.**
- Direct, professional, no sycophancy.
- No casual sign-offs ("sleep well", "good luck", etc.) — Eric finds them out of place.
- Acknowledge mistakes without self-flagellation.

### Phase 6 Step 9 Cleanup — The C.4 Transitional Duct Tape

Phase 6 Step 9 Cleanup is documented in:
- `docs/PHASE_6_STEP_9_CLEANUP_4_C4_TRANSACTION_RESULT.md`
- `docs/PHASE_6_STEP_9_CLEANUP_4_C5_FINAL_SWEEP.md`

C.4 migrated several Result classes (UserResult, BiometricResult, AuthResult, TransactionResult) toward a localizable error pattern. To avoid breaking consumers during migration, C.4 introduced a **transitional shape** where each result carried BOTH:
- `error: String?` — the legacy English error message
- `errorKey: SomeErrorKey?` — the new typed key

C.5 was the final localization sweep. Its Decision #9 explicitly **deferred** the collapse of `error:String?` to "a separate hygiene cleanup."

That separate hygiene cleanup is Phase A2, completed in this session.

## 9. Known Latent Issues

These are issues observed but not fixed. They are not blocking, but worth knowing.

**Hive corruption on cold start.** Hive 2.2.3 prints "Recovering corrupted box" on cold start. Masked, not actually breaking anything observable. Would need a Hive upgrade or storage rework to fix.

**AuthNotifier transient-null bug.** `_init()` in AuthNotifier calls `clearAll()` when it sees a transient null user, which can wipe legitimate session state during reauth. Workaround: cold restart. Real fix would be a state machine for auth init.

**App Check / Play Integrity 403.** On Note 9 debug builds, App Check returns 403 on Play Integrity attestation. This blocks cloud function calls in debug. Release builds work. Workaround: test cloud-function-dependent flows on release builds or another device.

**Pre-existing dead code.** Several unused helpers (`_generateUniqueWalletId`, `_validateKey`, `_generateReference`, `_generateTransactionId`, etc.) flagged by analyzer. Left alone — not introduced by recent work, removing would require verifying nothing reflects on them.
