## Where we are right now

Phase 6 (app-wide localization in English / French / Arabic) is functionally done for the user-facing screens. Tasks 1A (disputes screens localization) and 1B (pin screens localization) both shipped to `main` in May 2026. The app currently supports three languages with a first-launch picker, persistent language preference, and RTL layout for Arabic — all verified on a Galaxy Note 9.

A four-round investigation conducted on 2026-05-15 (transcribed in the session that produced this document) established the following baseline:

**The "Report Issue" button on the transaction details screen is broken in production.** When a user taps it, the app tries to navigate to a route called `/file-dispute` that is not registered with the router. The user lands on a generic "Page not found" screen with no back button.

**The screens needed to handle disputes already exist and are complete.** Two dispute screens (`FileDisputeScreen` and `RespondToDisputeScreen`) were built in Phase 2d, localized in Phase 6 Task 1A, and are production-shape forms with validation, submission, and localized error messages. They were never connected to the app's navigation system. That's the only thing missing for basic dispute filing to work end-to-end.

**The server-side dispute system works for the basic flow.** Four user-facing Cloud Functions exist and are deployed: `userFileDispute`, `userViewDispute`, `userGetMyDisputes`, `userRespondToDispute`. The admin investigation functions also exist. The current state machine is `filed → investigating → supervisor_review → manager_review → resolved | closed_stuck`.

**The Phase 5i upgrade is entirely ahead of us.** Thirteen new Cloud Functions are planned but none are written. The new state machine (with `solved`, `awaiting_release`, and other intermediate states), platform escrow wallets, two-person release flow, recovery watch rewrite, and stuck detection are all not started.

## The plan

We will do the work in four phases, in this order:

### Phase A — Close out Phase 6
Small mechanical cleanup items that have been deferred. Goal: declare Phase 6 fully done so we can stop carrying it.

### Phase B — Connect the broken Report Issue button to its screens
Small Flutter routing change. Goal: the button stops landing users on "Page not found". Users can file disputes through the existing system.

### Phase C — Build the Phase 5i upgrade, ordered by risk
The major work. Eric's preference is to tackle the riskiest pieces first while we have the most attention and recovery time, ending with the safest pieces. Subject to dependency constraints (see below).

### Phase D — Update the admin dashboard for Phase 5i
The web admin tools (separate codebase at `admin-dashboard/`) need parallel updates to operate the new flow.

## Phase A — Close out Phase 6

Two technical items plus one operational item.

### A1. The `withOpacity` deprecation sweep

Flutter has deprecated `withOpacity(X)` in favor of `withValues(alpha: X)`. The analyzer flags every instance. We know of at least five in the pin and dispute screens; a codebase-wide grep will find more. Probably 30–60 instances total.

This is mechanical find-and-replace. Suitable for delegating to the code agent. Estimated effort: 30–60 minutes including verification and smoke test.

### A2. The Result-type `error: String` field collapse

The handover documents flagged this as a transitional design in the codebase: certain Result types carry an `error: String` field that's intended to be temporary. Before we can plan the cleanup, we need to investigate the current use of this pattern to confirm the scope. May turn out to be a small fix; may turn out to be larger than expected; may turn out to be "leave it alone." Investigation comes first, then we decide.

### A3. Native-speaker review for French and Arabic (operational, not code)

All Phase 6 translations were machine-generated by Claude. The plan in `docs/PHASE_6_RESOLVED.md` Q2 calls for native speakers to do a quality review post-ship. Eric has Arabic-speaking contacts; French reviewer to be sourced. Money-related Arabic verbs are highest priority for review (إرسال, استرداد, تحصيل).

This is operational work, not coding. It can run in parallel with Phases B, C, D and does not block them. Tracking it here so it doesn't get forgotten.

### Definition of Phase 6 done

A1 and A2 complete and merged to `main`. A3 is tracked separately and does not gate the rest of the plan.

## Phase B — Wire up the orphan dispute screens

The smaller of the two Flutter changes. Estimated effort: under one hour including smoke test.

### What needs to happen

Three small additions to `lib/core/router/app_router.dart`:

1. Add `static const String fileDispute = '/file-dispute';` to the `AppRoutes` class
2. Add `static const String respondToDispute = '/respond-to-dispute';` to the `AppRoutes` class (or similar — exact path TBD when the work is done; should accept a `:disputeId` parameter for respond, no parameter for file since the data comes via `extra:`)
3. Add `GoRoute` entries for both routes
4. Add the two missing imports at the top of the file

`FileDisputeScreen`'s constructor already accepts `transactionId`, `transactionAmount`, `transactionCurrency`, and `recipientName` — the exact fields the existing Report Issue button passes via `context.push('/file-dispute', extra: {...})`. No screen changes needed.

`RespondToDisputeScreen`'s constructor accepts `disputeId`. Currently no caller pushes to it. Phase B would register the route, but no place in the app yet pushes to `/respond-to-dispute`. That second step ("how does the seller reach the respond screen?") is naturally a Phase C concern, because it requires the new tabbed `my_disputes` view that Phase 5i adds. Phase B just registers the route so it's available.

### Verification

After wiring:
1. On Note 9 device with a signed-in test user who has any outgoing send transaction less than 7 days old: tap Report Issue from the transaction details screen.
2. Confirm the `FileDisputeScreen` renders correctly in English, French, and Arabic.
3. Fill in the form, submit, confirm the dispute is created (visible via admin dashboard or `my_disputes_screen`).
4. Confirm the user can navigate back cleanly.

### Important caveat

Any dispute filed between Phase B shipping and Phase C completing goes through the **legacy** state machine (`filed → investigating → supervisor_review → manager_review → resolved`). It does not use the new Phase 5i escrow flow. Per `docs/PHASE_5I_SPEC.md` Section 11.1, legacy disputes stay on the legacy code path forever — they do not get migrated when Phase 5i lands. This is acceptable: interim disputes complete under the old rules, new disputes after Phase 5i deploy use the new rules.

If user volume during the interim is expected to be large, hide the button instead (described above as the alternative in Decision 1). Eric chose to wire it, accepting interim legacy disputes.

## Phase C — Phase 5i implementation, risk-first

This is the major work. Multi-session, spans backend and Flutter, money-handling-critical.

### Eric's stated preference

Tackle the riskiest pieces first, save the safer pieces for last. The reasoning: scary work deserves the most attention and recovery time; safe work can be hurried if needed.

### The constraint

The riskiest piece (the Recovery Watch rewrite per `docs/PHASE_5I_SPEC.md` Section 7.11) **depends on data structures and helper functions that don't exist yet**. Specifically, the rewritten Recovery Watch needs to read:

- The escrow wallet for the dispute's currency, created by a helper called `getOrCreateRecoveryWallet`
- A `sourceUid` field on each `wallet_debts` document, populated at the time the debt is created
- New states like `solved` and `awaiting_release` in the dispute document, used to know when to transition

None of these exist today. So a literal "Recovery Watch first" approach has nothing to call.

### How we'll honor "risk-first" given the constraint

A small foundations group MUST come first because nothing else can be written without it. Then the riskiest core work follows, then the safer additions, then the polish. The foundations themselves are very low risk — they're prerequisite infrastructure (helpers, rules, indexes) and don't move money.

### The four groups within Phase C, in order

#### Group C1 — Foundations (small, low risk, ~1 session)

These are prerequisites for everything else. No money movement.

- `getOrCreateRecoveryWallet` helper function (Section 7.1) — creates one wallet per currency at `wallets/dispute_recovery_<CURRENCY>`
- `generateClosingRemarks` helper function (Section 7.2) — text-only template engine for the closing remarks shown on closed disputes
- Firestore rules updates for the new escrow wallet collection and the new dispute fields (Section 8)
- Firestore composite indexes for the new queries that Phase 5i needs (Section 11)

Risk profile: **LOW.** None of this moves money. Helpers are pure functions or document-creation. Rules and indexes are configuration.

#### Group C2 — The risky core (multi-session, HIGH risk)

This is what Eric meant by "the riskiest piece." Everything in this group touches money flow or fundamentally changes how disputes are processed.

- Modify `userFileDispute` (Section 7.3) to populate the new dispute fields at filing time, place an opportunistic wallet hold on the buyer, and SMS the seller
- Add the new states to the backend state machine (`solved`, `awaiting_release`, `closed_returned`, `closed_buyer_cancelled`, `closed_partial_buyer_choice`, `stuck_pending_review`)
- Modify `adminManagerDecision` (Section 7.5) — the new decision outcomes (`buyer_owes_seller_full`, `buyer_owes_seller_partial`) and the new state transitions (`manager_review → solved` instead of `→ resolved`)
- Modify `adminSuperAdminDisputeDecision` (Section 7.6) — symmetric updates
- Add `adminProposeDisputeRelease` (Section 7.7)
- Add `adminConfirmDisputeRelease` (Section 7.8)
- Add `adminRejectDisputeRelease` (Section 7.9)
- Rewrite Recovery Watch (Section 7.11) — the spec calls this "the riskiest single change in Phase 5i — every line involves money"

Risk profile: **HIGH.** Every function in this group either moves money or changes the conditions under which money moves. Recovery Watch alone runs against many disputes at once on a schedule.

Recommended sub-ordering inside Group C2 (within the group, do the riskiest first per Eric's preference):
1. Recovery Watch rewrite — riskiest, do it while focus is highest
2. The two-person admin release flow (propose / confirm / reject)
3. `userFileDispute` and `adminManagerDecision` modifications
4. The new state machine additions

Recovery Watch is at the top of the sub-order because it's the spec-flagged highest-risk function. Doing it first means we get the scariest dependency out of the way; the foundations from Group C1 give it what it needs to read, and the admin release flow uses it for its `awaiting_release → closed` transition.

#### Group C3 — User-facing additions (~1 session, MEDIUM risk)

These all trigger admin or release flows; they don't directly move money themselves but they create the conditions for money movement.

- `userRequestPartialRelease` (Section 7.10) — buyer asks to take what's collected, choosing close_after vs continue_collecting
- `userCancelDispute` (Section 7.10) — buyer requests cancellation, requires admin approval
- `adminConfirmCancellation` (Section 7.10) — admin approves the user's cancel request, refunds any held money
- Flutter UI for both: action buttons in `dispute_detail_screen` plus localization strings in en/fr/ar
- Flutter UI for the seller's response form — either wire up the existing `RespondToDisputeScreen` (Section 9.2 path) or integrate the form into `dispute_detail_screen` (Section 9.1 path). Implementer chooses during this group.

Risk profile: **MEDIUM.** Each function triggers downstream effects but doesn't directly move money in its own transaction.

#### Group C4 — Polish and monitoring (~1 session, LOW risk)

The safest pieces. Eric explicitly wanted these last.

- Scheduled function `disputeAccountClosureCheckScheduled` (Section 7.12) — runs daily, moves disputes whose source's account has been closed for 3 days into `stuck_pending_review`
- Scheduled function `disputeNoProgressCheckScheduled` (Section 7.12) — runs daily, moves disputes with no recovery progress in 90 days into `stuck_pending_review`
- `disputeReleaseProposalExpiryScheduled` (Section 7.15) — auto-rejects release proposals that go unconfirmed
- `adminMarkDisputeAccountClosed` (Section 7.13) — manual override to mark an account closed
- `adminExportDisputeEvidencePackage` (Section 7.14) — read-only PDF/zip export for closed disputes
- Tabbed `my_disputes_screen` per `docs/PHASE_5I_RESOLVED.md` Q4 — "Filed by me" / "Filed against me" tabs with count badges and unread red dot
- Updated `dispute_detail_screen` rendering for new statuses (progress bar for `solved`, "Awaiting verification" banner for `awaiting_release`, closing remarks for closed states)
- Per-role action panels in `dispute_detail_screen` (buyer sees cancel + request partial release; seller sees response form)

Risk profile: **LOW.** Scheduled functions handle state transitions but don't directly move money. UI changes are presentational. Evidence export is read-only.

### Phase C deployment

Note that **implementation order ≠ deployment order.** Per `docs/PHASE_5I_SPEC.md` Section 12, deployment order is binding and follows a different sequence (rules first, then indexes, then helpers, then admin read-only functions, then seller response, then buyer self-service, then admin cancellation, then release flow, then state machine cutover, then Recovery Watch rewrite last).

What this means in practice: we can build Recovery Watch first (Group C2 sub-step 1) while it's still under feature flag / commented out / behind a kill switch, get it reviewed and tested in isolation, then deploy it last per the spec's deploy order. Phase C2 is implementation; deployment per Section 12 is a separate orchestration that happens at the end of Phase C as a coordinated rollout.

The spec's binding deploy order is binding for a reason: each step in production should be safely revertable, and earlier production-deployed steps must not depend on later ones. We respect it at deploy time regardless of implementation order.

## Phase D — Admin dashboard updates

The web admin dashboard (separate codebase at `admin-dashboard/`) needs parallel updates so the admin team can operate the new flow.

### What needs to happen

Per `docs/PHASE_5I_SPEC.md` Section 10:

- Locate and update the existing disputes page (likely `DisputesPage.jsx` or `DisputesListPage.jsx`) to filter on the new statuses
- Add columns for `decisionDirection`, `amountInEscrow / amountOwed`, and release-proposal status
- For each new status, add appropriate admin actions to the dispute detail page (Propose Release modal for `awaiting_release`, Confirm Cancellation form for `solved + cancellationRequested`, Process Partial Release link for `solved + partialReleaseRequested`, Export Evidence Package button for closed states)
- New page: "Awaiting Release Queue" — filtering disputes in `awaiting_release` state, showing pending proposals (admin can confirm/reject) and disputes with no proposal yet (admin can propose)
- `NotificationBanner.jsx` — add a watcher for release proposals awaiting the current admin's confirmation (in `awaiting_release` with `releaseProposal` set and not proposed by current admin)

### Why D comes after C

Building D in parallel with C would mean the admin tools change before the backend they operate against. Risk of admin UI calling functions that don't exist yet. Doing D after C means the backend is stable when admin UI consumes it.

The trade-off is that during Phase C, the admin team can't operate the new flow until D is done. This is acceptable because Phase 5i has a coordinated deploy at the end — nothing in C is user-visible until everything is ready.

## Workflow rules (non-negotiable)

These caused trouble earlier in the project. Treat as locked-in.

- **Investigate before guessing.** `grep`, `cat`, `sed` first. Never declare success on stale logs. When uncertain, look at the actual code before making any claim about it.
- **Plain explanations on request.** If asked to explain in plain English, do so — no jargon, no abbreviations. Future sessions: respect the same standard.
- **Eric runs commands locally and pastes output.** Claude doesn't have filesystem access to Eric's Mac. Drafts of code-agent instructions go in `/mnt/user-data/outputs/` for Eric to copy.
- **Git hygiene.** NEVER `git commit -a` or `git add .`. Always explicit `git add <file>`. Single-line commit messages only — zsh has quote-handling issues with multi-line messages containing parens or other shell-special characters.
- **No verbal recommendations until full understanding.** Eric's stated rule: investigate, then surface decisions, then let Eric pick. Don't pre-pick options. Don't propose paths until the investigation is complete.
- **Firestore production indexes.** Production has a `transactions: createdAt DESC + type ASC` index that is not in local `firestore.indexes.json`. If the CLI prompts to delete it, always answer No.
- **Smoke-test protocol.** Required before every commit. Cold restart only (quit and rerun, OR swipe away from recents). Hot restart (`R`) is NOT valid for persistence testing.
- **Test device:** Galaxy Note 9 (SM N960F, Android 10, device ID `254c35c4c21c7ece`). Always pass `-d 254c35c4c21c7ece` to `flutter run`. The Android emulator is not a substitute — it has App Check / Play Integrity rejection issues that the Note 9 does not.
